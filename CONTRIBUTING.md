# Contributing

## Zasady pracy
- Każda większa zmiana powinna być wykonywana na osobnym branchu.
- Nazewnictwo branchy:
  - `feature/nazwa-funkcji`
  - `fix/nazwa-poprawki`
  - `docs/nazwa-zmiany`
- Kod mergujemy do `main` po sprawdzeniu przez pozostałych członków zespołu.
- Większe zmiany powinny być odnotowane w dokumentacji w katalogu `docs/`.

## Commit messages
Zalecane są krótkie i opisowe komunikaty, np.:
- `add initial project documentation`
- `create PRD and architecture docs`
- `add upload module plan`
- `update implementation roadmap`

## Zasady dotyczące danych
- Nie dodajemy prawdziwych faktur zawierających dane wrażliwe.
- W katalogu `data/sample_data/` przechowujemy tylko dane testowe lub zanonimizowane.

## Styl pracy zespołowej
- Najpierw ustalamy zakres funkcji w `docs/TODO.md`.
- Każdy moduł powinien mieć właściciela odpowiedzialnego za integrację.
- Przed prezentacją projektu aktualizujemy README i dokumentację.
